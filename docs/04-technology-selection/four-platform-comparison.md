## 一、执行摘要

**结论**：推荐以 **Dify** 作为 AIDevOps 平台的核心底座，Open Harness 作为技术参考值得关注但尚不适用于生产环境。

| 维度 | pi | Dify | Mastra | Open Harness |
|------|:---:|:----:|:------:|:-----------:|
| 综合加权得分 | 2.15 | **3.35** | 3.30 | 1.75 |
| 推荐优先级 | ❌ 不推荐 | ✅ 首选 | ⚠️ 备选 | 🔍 技术参考 |

**Open Harness 定位**：
Open Harness 的 VCR 录制/回放、事件溯源、时间旅行调试等架构创新值得学习，但其目前仅有 12 stars、v0.1.1、单 Provider（仅 Claude）、无 MCP/无 Hook/无 Skill/无多租户——距离生产可用还有显著差距。

## 四、逐维评估

### D1：Agent 对话核心能力（权重 20%）

#### pi — 4/5
状态化事件驱动循环，JSONL 会话管理完善，对话引擎质量高。

#### Dify — 3/5
Agent 核心 2.3K LOC，FC + CoT 策略，7 种 AppMode，graphon DAG 引擎（21 种节点）。核心较薄，需扩展增强。

#### Mastra — 5/5
Agent 核心 7,086 行代码，StreamProcessor、MessageList 机制完整，对话状态管理原生实现。

#### Open Harness — 3/5
核心 engine/agent 仅 191 LOC，结构化输出（Zod schema）完善。PhaseWorkflow 状态机模式支持对话路由，但自主决策能力有限，主要依赖 Phase 路由分发任务。

## 六、Open Harness 深度分析

### 6.1 核心架构

Open Harness 采用**事件溯源（Event Sourcing）**架构，核心引擎分层：

```
agent (191 LOC)     → 对话状态管理
workflow (397 LOC)  → 工作流编排
phase (224 LOC)     → 状态机阶段
runtime (1022 LOC)  → 运行时核心
```

两种工作流模式：
- **SimpleWorkflow**：单 Agent 循环，适合简单任务
- **PhaseWorkflow**：状态机模式，支持分支、并行（forEach）、人工介入（HITL）

### 6.2 创新特性

| 特性 | 说明 | 对 AIDevOps 的借鉴价值 |
|------|------|----------------------|
| **VCR 录制** | 录制 Provider 调用，回放无需 API 消耗 | 测试/调试场景有价值 |
| **时间旅行** | GET 任意事件位置的状态 | 调试复杂对话流程 |
| **Session Fork** | 克隆并分支会话 | 并行探索/对比实验 |
| **HITL** | 人工审批/选择节点 | 需要人工确认的 DevOps 流程 |

### 6.3 关键差距

| 缺失项 | 影响 | 弥补难度 |
|--------|------|---------|
| 无 MCP | 无法对接现有工具生态 | 高（需从零实现） |
| 无 Hook 系统 | 无法动态拦截/增强 | 高（需重设计） |
| 无 Skill | 无法动态加载知识 | 高（需从零实现） |
| 无 SubAgent | 无法编排外部 Agent | 高（需从零实现） |
| 无多租户 | 无法多用户隔离 | 高（需从零实现） |
| 单 Provider | 仅支持 Claude | 高（Provider 架构受限） |
| 无 Web UI | 仅有 API + React Hooks | 高（需从零构建） |

### 6.4 生产就绪度评估

**当前版本不推荐用于生产环境**：

1. **规模验证缺失**：28K LOC vs Mastra 的 74MB / 2,900 文件，未经大规模生产验证
2. **社区极小**：12 stars，5 贡献者，无企业用户背书
3. **活跃度下降**：最后提交 3+ 月前，v0.1.1 后无后续 release
4. **安全缺口**：API 无认证，CORS 完全开放
5. **功能缺失**：6 项核心需求完全缺失（见上表）

**技术参考价值**：事件溯源、VCR 录制、时间旅行等设计模式值得借鉴，可考虑在未来版本吸收其实践。

## 八、最终推荐

### 8.1 推荐方案：**Dify 渐进式改造**

**核心策略**：以 Dify 作为完整平台底座，裁剪不需要的模块，增强 Agent 核心能力，补充 Skill 动态注入和外部 Agent 调度。

**执行路径**：

```
阶段 1（月 1）：裁剪瘦身
  - 裁向量库（Weaviate→pgvector）、裁无关 AppMode
  - 12 容器 → 7 容器，460K → ~150K LOC

阶段 2（月 2）：业务定制 + Skill 增强
  - 加 AgentCenter 业务页面（看板/概览/上下文设置）
  - 在 _organize_prompt_messages() 层加 Skill 动态注入（4-5 人天）

阶段 3（月 3）：Agent 核心增强
  - 扩展 core/agent/，增加 Hook 机制
  - 设计 SubAgent 接口，对接 ACP 协议
  - Agent 核心从 2.3K LOC → ~5K LOC

阶段 4（月 4）：生产加固
  - 安全审计（沙箱 per-tenant 隔离）
  - 审计追溯增强
  - 监控告警
```

### 8.2 备选方案：Mastra 从零建平台

如团队更看重 Agent 核心质量，且愿意投入更多时间构建平台：
- 优点：Agent 核心成熟（7K LOC）、SubAgent 原生支持、Hook 系统、TypeScript 全栈
- 缺点：需从零构建 Web 平台（60+ 人天）和多租户（10-15 人天）
- 策略：4 个月出可用版本，长期架构质量更优

### 8.3 技术参考方案：Open Harness

Open Harness 的事件溯源架构、VCR 录制/回放、时间旅行调试等创新设计值得借鉴，但不建议作为底座直接使用。建议：
- 关注其后续发展，6-12 个月后再评估
- 借鉴其事件溯源设计优化 AIDevOps 的会话管理和调试能力
- 避免在其早期生态上投入建设

### 8.4 不推荐方案：pi

pi 不适合作为 AIDevOps 平台底座，原因：
1. 无 MCP 支持，与工具生态需求根本矛盾
2. 无 Web UI，与平台化需求根本矛盾
3. 单用户架构，多租户改造代价最高
4. 建议将 pi 作为**嵌入式对话引擎**模块集成，而非平台底座
